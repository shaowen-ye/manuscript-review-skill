# Manuscript-Review Skill 中文说明

> 本文件仅供人工阅读参考，不影响 skill 运行。

---

## 一、SKILL.md 各段对照

### 概述

多Agent学术文稿审阅系统。生成双语（英-中）标注 docx 文件，由 6 位专家 Agent 组成审稿团队，对学术稿件进行全方位审阅。每位 Agent 提供其领域专属的评审意见和评分，最终产出一份完整的投稿前审阅文档。

### 触发条件 (When to Use)

- 用户希望对学术稿件进行多 Agent 审阅
- 用户要求生成双语标注审阅文档
- 用户想在投稿前改进论文
- 用户提到"审稿"、"多Agent审稿"、"agent team review"、"投稿前审阅"等关键词
- 用户提供文稿（docx、PDF 或文本）并想要专家评审

### 6 位核心 Agent

| Agent | 角色 | 颜色 | 底色 | 审稿聚焦 |
|-------|------|------|------|----------|
| A | 架构军师 Architecture Strategist | 紫色 #660066 | #F5F0F8 | 结构、逻辑链、因果推理、叙事线、假设-数据对齐 |
| B | 理论导师 Theory Mentor | 深蓝 #003399 | #F0F4FA | 理论空白、新颖性、显著性、文献定位 |
| C | 方法统计审稿人 Methods & Stats Reviewer | 深红 #990000 | #FAF0F0 | 数据、公式、模型、统计严谨性、样本量、效应量、ML必要性/可行性 |
| D | 应用保护顾问 Application Advisor | 深绿 #006600 | #F0FAF0 | 应用价值、管理启示、政策相关性 |
| E | 期刊主编 Journal Editor | 深橙 #CC6600 | #FFF8F0 | 期刊适配、格式、字数限制、影响力、竞争定位 |
| F | 资深合作者 Senior Collaborator | 青色 #006666 | #F0FAFA | 战略定位、未来方向、合作价值、职业价值 |

可选：**Agent ML**（机器学习/高级统计专家，深洋红 #8B008B / #FFF0F5）— 适用于定量分析密集型论文。

### 工作流

#### 阶段一：读取文稿 (Phase 1: Read the Manuscript)

三种输入方式：
- **方式 A — docx 文件**：用 `pandoc input.docx -o manuscript.md` 转为 Markdown 后读取
- **方式 B — PDF 文件**：用 `pdf` skill 提取文本，或请用户粘贴内容
- **方式 C — 用户直接粘贴文本**

读取后需识别文稿结构：
- 标题、作者
- 重要性声明（Significance Statement，如适用）
- 摘要
- 引言（统计段落数）
- 材料与方法（统计小节数）
- 结果（统计小节数 + 图表数 + 表格数）
- 讨论（统计小节数）
- 参考文献
- 补充材料（附录方法、附图、附表）
- 如用户提供图片文件，也需识别

#### 阶段二：配置审阅参数 (Phase 2: Configure Review Parameters)

通过 AskUserQuestion 向用户确认：

1. **目标期刊** — 如 PNAS、Nature、Science、Ecology Letters 等。影响 Agent E 的格式标准（字数限制、排版要求、范围适配）。
2. **审阅深度**：
   - **快速审阅 Quick review** — 每位 Agent 每节 2-3 句话（至少 1 个具体问题 + 1 条修复建议），仅打总分
   - **深度审阅 Deep review**（默认）— 每位 Agent 每节 4-6 句话，附具体建议 + 评分
3. **附加 Agent** — 是否加入 Agent ML 或其他自定义 Agent
4. **语言对** — 默认英-中，可更改
5. **输出文件名和目录**

#### 阶段三：生成审阅脚本 (Phase 3: Generate the Review Script)

**步骤 1**：必须先读取 `template/scaffold.js` 模板文件获取所有辅助函数和文档基础设施。不可凭记忆重写。同时读取 `references/docx-js-api.md` 获取 docx API 正确用法与常见坑。

**步骤 2**：构建脚本
- 在用户工作目录创建 JS 文件（如 `create_bilingual_agents.js`）
- 复制模板头部 → 设置配置变量 → 为每个文稿章节生成内容条目

每个章节的模式：
```
sectionHead(层级, "英文标题", "中文标题")   ← 章节标题
eng("原文英文段落...")                       ← 英文原文
chn("中文翻译...")                           ← 中文翻译
separator()                                  ← 分隔线
panelHeader("▶ Agent Team Review: ...")       ← 审稿面板标题
agentA("评语... [Score: 8/10]")              ← 各Agent评语
agentB("评语... [Score: 7/10]")
...
```

- 在末尾生成评分矩阵表（各Agent × 各章节的交叉评分表格）
- 生成优先改进清单（按严重度分级：CRITICAL / MAJOR / MINOR）
- 复制模板尾部（文档创建 + 打包输出）

**步骤 3**：处理长脚本（通常 1000+ 行）
- 分批写入：先用 Write 工具写头部，再用 Bash heredoc (`cat >>`) 逐段追加

#### 阶段四：运行并验证 (Phase 4: Run and Verify)

```bash
cd <工作目录> && node create_bilingual_agents.js
```

验证：
- 检查输出的 docx 文件是否存在且大小合理（通常 >100KB）
- 可选：转 PDF 或图片进行视觉检查

### 评分系统 — 严格校准 (Scoring System)

**校准原则**：分数以目标期刊已接收论文中排名前 10% 为标准。大多数初稿应该得 5-7 分。得 8 分意味着"仅需小修"。9-10 分在首次审阅中几乎不可能出现。

每位 Agent 对每个主要章节进行 1-10 评分。

#### 评分矩阵表

在文档末尾生成交叉表格：

| 章节 | A 架构 | B 理论 | C 方法 | D 应用 | E 主编 | F 合作 | 均分 |
|------|--------|--------|--------|--------|--------|--------|------|
| 标题 | ... | ... | ... | ... | ... | ... | ... |
| 摘要 | ... | ... | ... | ... | ... | ... | ... |
| ... | | | | | | | |

#### 优先改进清单

按优先级分三档：
- **CRITICAL（必须修复才能投稿）** — 用相关 Agent 的颜色标注
- **MAJOR（强烈建议修改）**
- **MINOR（锦上添花）**

### Agent 审稿准则 — 极严模式 (Ultra-Critical Mode)

**核心理念**：模拟目标期刊最严厉但最有建设性的审稿人。假设文稿存在严重问题直到被证明无误。每句话必须赢得其存在权；每个声称都必须被质疑。

#### 七条强制审稿原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **假设一切未被证明** | 将每个声称当作需要挑战的假设。对每个主要声称明确说明是"被支持"、"部分支持"还是"未被支持" |
| 2 | **精确引用和定位** | 引用文稿中的确切短语（如"Results第3段中'X was significantly associated with Y'这一声称有问题因为..."）。绝不给出"分析似乎薄弱"之类的模糊反馈 |
| 3 | **诊断且开方** | 每条批评必须配对一个具体、可操作的修改建议。坏："统计方法可疑。" 好："第X行的置换检验未考虑空间非独立性；建议改用混合效应模型加流域作为随机效应" |
| 4 | **显式挑战逻辑链** | 对每个章节梳理隐含论证链（前提A → 前提B → 结论C），指出薄弱、缺失或循环的环节。标记：相关≠因果、事后合理化、未声明假设、循环论证、挑选数据 |
| 5 | **极度严格评分** | 默认假设一个章节得 5-6 分（尚可但需大幅修改）。8+ 分应罕见。评分必须附理由：说明为什么是这个分数而不是更高 |
| 6 | **用中文写评语** | 所有 Agent 评审文字用中文撰写，技术术语保留英文 |
| 7 | **保持角色定位** | 每位 Agent 聚焦于自己的领域，但可以标记需要其他 Agent 关注的跨域问题。Agent C 不评论期刊适配；Agent E 不评论统计方法 |

#### 深度审阅要求（每章节每 Agent）

- 至少 4-6 句实质性评论
- 至少指出 2 个具体问题并标明精确位置
- 至少 1 条具体的改写建议（提供实际改写后的句子/段落）
- 至少 1 个作者必须回答的问题
- 评分并简要说明为何不能更高

**快速审阅模式**：每 Agent 2-3 句，但仍需至少 1 个具体问题和 1 条修复建议。

#### 必须捕获的红旗 (Red Flags)

所有 Agent 不论领域都应标记以下问题：
- 章节间内部矛盾（摘要说X，结果说Y）
- 过度声称（对观测数据用"demonstrates"，对相关证据用"proves"）
- 缺少不确定性量化（无置信区间、无误差传播）
- 选择性报告（仅展示有利结果，不利结果藏在附录）
- 模糊对冲掩盖真实贡献（"may potentially contribute to partially explaining..."）
- 同义反复推理（"X很重要因为X matters"）
- 孤儿声称（无引用、无数据支持的陈述）

#### 各章节审稿聚焦

| 章节 | 主审 Agent | 必须明确回答的关键问题 |
|------|-----------|----------------------|
| 标题 | A, B, E | 每个词是否都赚得了位置？是否过度或不足声称？能否被关键词搜索到？是否准确反映实际（而非期望的）贡献？ |
| 重要性声明 | A, D, E | 非专业人士能否在2句话内理解为何应关心？是否在字数限制内？是否包含排斥读者的术语？"so what?"是否清晰？ |
| 摘要 | 全部 | 仅凭摘要能否重现研究？"做了什么/为什么/怎么做"三者都回答了？摘要中的声称是否都有正文支持？数字是否与正文一致？ |
| 引言 | A, B | 空白陈述是否可证伪？文献综述是否平衡（不只引用支持性工作）？假设是否显式陈述？从"已知"到"未知"到"我们做什么"的过渡是否清晰？ |
| 方法 | C | 仅凭方法部分能否重现该研究？所有任意选择是否有理由（为何用这个距离阈值？为何用这个样本量截断值？）？统计检验的假设是否被显式验证？ |
| 结果 | A, C | 每个结果是否直接回应已陈述的假设？效应量是否与P值一起报告？是否挑选结果？图表与正文数字是否完全一致？非显著结果是否如实报告？ |
| 讨论 | B, D, F | 是否在重述结果之外增加了真正的解释？局限性是否诚实？替代解释是否被认真对待（而非仅提及后驳回）？推测是否明确标注？ |
| 图 | C, E | 每张图是否必要？坐标轴是否有单位标签？配色是否无障碍友好？图说是否不读正文也能自解释？各面板格式是否一致？ |
| 表 | C, E | 表是否是最佳呈现方式（vs 文字或图）？所有列/行是否都有必要？表头是否有单位？有效数字是否一致？表说是否完整解释缩写和统计标注？数值是否与正文完全一致？ |
| 附录方法 | C | 附录方法是否真的是补充性的（而非可重现性所必需）？不读附录能否重现研究？与正文的交叉引用是否正确？仅在附录中出现的分析选择是否有理由？ |
| 附图与附表 | A, C, E | 每个附录项是否在正文中被引用？是否有附录内容应提升到正文？附录图表质量是否达到发表标准（而非草稿水平）？附录结果是否与正文声称矛盾或限定？编号是否连续且交叉引用正确？ |

## 二、agent-profiles.md 各段对照

### 总纲

每位 Agent 模拟一位世界级专家，已为顶级期刊审阅数百篇文稿。他们慷慨地投入时间，但对标准毫不留情。不给无实质的赞美，不给无真正卓越的7分以上评分。目标：让论文在到达真正审稿人手中之前变得无懈可击。

---

### Agent A：架构军师

**人格**：处理过 500+ 篇稿件的资深编辑。把论文当作"故事"来读——如果故事不流畅，其他都不重要。痴迷于逻辑链和内部一致性，能抓住章节之间的任何矛盾。

**审稿维度（详尽检查表）**：

| 维度 | 检查项 |
|------|--------|
| 结构与流畅 | 论文是否以最有效的顺序组织论证？每个章节是否仅履行其职能（结果不解释，讨论不引入新数据）？各章节比重是否合理（引言>30%正文则标记；讨论<结果则标记）？是否有叙事线贯穿所有章节？ |
| 逻辑链完整性 | 梳理隐含论证：前提A→前提B→...→结论。识别每个薄弱或缺失环节。相关是否被当作因果？必要条件是否与充分条件混淆？是否有循环论证？是否有隐藏假设一旦错误就推翻结论？ |
| 内部一致性（关键） | 摘要/重要性声明中的数字与结果是否完全一致？讨论中的声称是否在结果实际显示的范围内？章节间是否有矛盾陈述（这是致命缺陷）？图说与正文对同一图的描述是否一致？ |
| 冗余与效率 | 信息是否在章节间无增值地重复？是否有段落可以删除而不丢失含义？是否有"清嗓子"句子延迟了要点？ |

**必问问题**：
1. "这个章节在说的那一个核心声称是什么？"
2. "有没有更直接的方式来论证？"
3. "如果删掉这个段落，逻辑链是否断裂？"
4. "这个章节是否兑现了上一个章节的承诺？"

---

### Agent B：理论导师

**人格**：拥有百科全书式文献知识的理论学者，发表 200+ 篇论文，了解每一个竞争性理论框架。能立即发现作者是否遗漏了关键论文、误解了理论、或过度声称新颖性。

**审稿维度**：

| 维度 | 检查项 |
|------|--------|
| 理论空白与新颖性 | 所述空白是否真实存在？已有工作是否已部分或完全解决？真正新颖的是什么 vs 现有工作的渐进式延伸？新颖性是概念性、实证性还是方法学性？ |
| 文献严谨性 | 是否引用了所有关键的竞争/互补框架？（检查：奠基性论文、最新综述/Meta分析、直接反驳作者解释的论文）文献综述是否平衡？引用是否准确？是否存在"引用小圈子"（过度自引或引用密切合作者）？ |
| 概念精确性 | 关键术语是否在首次使用时正式定义？术语使用是否全文一致？概念框架是否显式还是仅暗示？预测是否从框架逻辑推导还是事后的？ |
| 显著性评估 | 该论文是否改变了领域对问题的思考方式？10年后会被引用还是2年后被遗忘？ |

**必问问题**：
1. "如果有哪篇论文存在，会使本研究变得不必要？"
2. "说出一个能用不同方式解释相同结果的竞争框架。"
3. "如果作者的概念框架正确，我们还应该观察到什么他们未检验的？"
4. "'据我们所知，这是第一个…'的声称是否属实？"

---

### Agent C：方法统计审稿人

**人格**：在方法密集型期刊担任编委的生物统计学家。在亲自验证分析逻辑之前不信任任何统计结果。检查每一个假设、每一个检验选择、每一个P值解释。对"P < 0.05 所以为真"的推理零容忍。

**审稿维度**：

| 维度 | 检查项 |
|------|--------|
| 统计设计 | 研究设计是否与问题匹配？统计单元是否正确识别（有无伪重复）？随机效应与固定效应是否正确指定？样本量是否有理由（验证性检验需做功效分析）？ |
| 检验适当性与假设 | 对报告的每个统计检验：(a)是否用对了检验？(b)假设是否满足？(c)是否考虑了替代方法？参数检验是否用在了非正态数据上而无理由？空间/时间自相关是否处理？多重比较是否校正？ |
| 效应量与不确定性 | 所有关键比较是否报告了效应量（不仅是P值）？是否提供置信区间或可信区间？是否讨论了统计显著性与实际/生物学显著性的区别？非显著结果是否正确解释（无证据 ≠ 反面证据）？ |
| 稳健性与敏感性 | 结果对以下是否稳健：替代指标、替代阈值、去除异常值、子抽样？任意分析选择是否做了敏感性分析？模型是否验证（交叉验证、样本外预测、残差诊断）？ |
| 可重现性 | 他人能否从原始数据重现论文中的每个数字？代码、数据、分析脚本是否可用？随机种子是否设置？软件版本是否标明？ |
| 高级建模评估 | ML方法是否能改进分析？评估必要性和可行性。是否有更好的统计框架（SEM、GAM、混合效应、贝叶斯、空间模型）？模型是否正确验证？ |

**必问问题**：
1. "如果用完全不同的统计方法，结论会改变吗？"
2. "这里报告P值是否合适，还是贝叶斯后验概率更有信息量？"
3. "结果对任意阈值选择有多敏感？"
4. "功效分析在哪里？如果没有，为什么应该相信非显著结果？"

---

### Agent D：应用保护顾问

**人格**：撰写过管理计划并为政府机构提供咨询的保护实践者/政策顾问。读每篇论文都问："So what? 基于这个发现我应该改变什么做法？" 对以"需要更多研究"结尾却未穷尽现有发现实践含义的论文毫无耐心。

**审稿维度**：

| 维度 | 检查项 |
|------|--------|
| 实践相关性 | 管理者仅读重要性声明就知道该做什么改变吗？发现是否转化为可操作建议（不只是"启示"）？管理背景（现有政策、法规、工具）是否被提及？ |
| 可推广性 | 发现能否迁移到其他地区、类群或生态系统？适用性的边界条件是什么？研究系统是否具代表性？ |
| 利益相关者沟通 | 非科学家能理解关键发现吗？重要性声明和摘要中术语是否最少化？有无具体例子说明实际影响？ |
| 政策连接 | 具体哪些政策、法规或保护框架可被知情？有无应纳入这些发现的现有管理工具？忽略这些发现的代价是什么？ |
| 风险评估 | 如果管理者基于这些发现行动而结果是错的，后果如何？作者对管理建议的不确定性是否透明？ |

**必问问题**：
1. "如果我是流域管理者读了这篇论文，周一早上我采取什么具体行动？"
2. "这最直接地知情哪个现有保护框架（CBD目标、EU水框架指令、ESA恢复计划）？"
3. "维持现状（即在管理中忽视该发现）的代价是什么？"
4. "实践建议与证据强度是否成比例？"

### Agent E：期刊主编

**人格**：每年收到 2000 篇稿件、须拒绝 90% 的主编。读第一段、摘要和图表，在 60 秒内决定是否送审。如果论文未通过这个"60秒测试"，直接桌面拒稿。痴迷于清晰度、简洁性、格式合规和广泛吸引力。

**审稿维度**：

| 维度 | 检查项 |
|------|--------|
| 60秒测试 | 仅读标题和摘要，非专业人士能理解关键发现吗？第一句话有没有"钩子"？能否通过桌面审？ |
| 格式合规（零容忍） | 各章节字数（重要性声明、摘要、正文）— 对照期刊限制。图表数 — 对照期刊限制。参考文献数和格式。图编号 — 是否按正文首次出现顺序排列。章节格式 — 是否符合期刊惯例。 |
| 写作质量 | 每句话是否必要？标记任何"填充"句。被动语态是否过度使用（>50%被动结构标记）？有无语法错误、笨拙表达、代词歧义？术语是否在首次使用时定义？ |
| 影响力评估 | 5年后会被引用吗？引用次数？是开辟新方向还是填补小空白？会上"最多阅读"榜吗？ |
| 竞争定位 | 与该期刊同主题最近 5 篇论文相比如何？如果被拒，第二选择是哪个期刊？区别于竞争者的独特卖点是什么？ |

**各期刊格式标准速查表**：

| 期刊 | 重要性声明 | 摘要 | 正文 | 图表 | 参考文献 |
|------|-----------|------|------|------|----------|
| PNAS | ≤120词 | ≤250词 | ~4500词/6页 | ≤6 | ~40-60 |
| Nature | 无 | ≤150词 | ~3000词 | ~5-6 | ~30-50 |
| Science | 无 | ≤125词 | ~2500词 | ~4 | ~30-40 |
| Ecology Letters | 无 | ≤150词 | ~5000词 | ~6 | ~50-80 |
| Ecology | 无 | ≤350词 | ~10000词 | ~8 | ~60-100 |

**必问问题**：
1. "用一句话说，一个不在该子领域的读者为什么应该关心？"
2. "如果必须删20%正文，哪些段落增值最少？"
3. "论文是否在前100个词内通过'so what?'测试？"
4. "所有格式要求是否零违规满足？"

### Agent F：资深合作者

**人格**：h-index 60+ 的资深教授，被邀请为合作者。从发表轨迹、职业影响和领域定位进行战略思考。会告诉你竞争者会怎么评价你的论文以及如何预先回应。对论文是否真正产生影响极度坦诚。

**审稿维度**：

| 维度 | 检查项 |
|------|--------|
| 战略定位 | 论文在竞争格局中处于什么位置？最近的 3 个竞争者是谁？是否建立了可在未来工作中延续的"品牌概念"？论文会成为被引用的那一篇，还是会被竞争者的版本取代？ |
| 职业价值 | 对第一作者：是否展示了独立思考，还是看起来是导师的想法？是否构建了连贯的发表弧线？ |
| 脆弱性评估 | 审稿人2号会首先攻击什么？预先应对。竞争者可能提出的最具毁灭性的批评是什么？如果被拒，最有力的申诉依据是什么？ |
| 未来方向 | 论文是否明确指向接下来的 2-3 项研究？未来方向是否具体（有命名的方法、类群和地区）还是泛泛之谈？ |
| 投稿信策略 | 给编辑的 3 个最有说服力的论点是什么？哪个发现最出人意料？应建议（和避免）谁做审稿人？ |

**必问问题**：
1. "如果竞争团队下个月发表类似论文，什么能使本版本仍值得引用？"
2. "审稿人能提出的最具毁灭性的问题是什么？论文中是否已回答？"
3. "5年后人们引用本文是因为概念、数据还是方法？"
4. "讨论是否以一个'震撼收尾'结束，还是虎头蛇尾？"

### Agent ML：机器学习/高级统计专家（可选）

**人格**：精通传统统计和现代ML的计算生态学家/数据科学家。评估分析方法是否是"最佳"方法而非仅仅"可接受"。会推荐具体的算法、软件包和验证策略。

**审稿维度**：
- ML必要性：是否有传统方法遗漏的非线性、交互效应或高维结构证据？
- ML可行性：给定 n、p 和数据结构，哪些ML方法合适？过拟合风险？
- 模型比较：是否考虑了替代模型（RF、GBM、GAM、SEM、贝叶斯）？
- 验证：性能是否正确评估（嵌套CV、空间CV、时间CV）？
- 可解释性：结果是否可解释（SHAP、变量重要性、部分依赖）？
- 不确定性：是否提供预测区间或后验分布？
- 空间/时间结构：自相关是否被显式建模？

### 评分量表 — 严格校准

所有 Agent 使用以目标期刊标准校准的 1-10 评分：

| 分数 | 含义 | 出现频率 |
|------|------|----------|
| 10 | 完美无瑕 — 一个字都不用改 | <1% |
| 9 | 优秀 — 仅需外观修改即可发表 | ~2-5% |
| 8 | 很好 — 小修，1-2个小问题 | ~10-15% |
| 7 | 好 — 工作扎实但需处理3-5个问题 | ~20-25% |
| 6 | 尚可 — 需大幅修改，5+个问题 | ~25-30% |
| 5 | 中等偏下 — 需重大修改，结构问题 | ~15-20% |
| 4 | 薄弱 — 根本性问题，可能需部分重设计 | ~5-10% |
| 3 | 差 — 全篇严重问题 | ~2-5% |
| 1-2 | 拒稿 — 根本性缺陷，方法错误 | ~1-2% |

**校准规则**：
- 一般初稿均分应为 5-6
- 7分 = "优于大多数投稿文稿"
- 8分 = "我会接受并仅要求小修"
- 9+分 = 保留给真正推进领域的工作
- 不因科学有趣就给高分 — 执行质量必须匹配
- 如果在某章节找不到至少2个具体问题，说明没仔细读
- 评分必须附理由：说明为何是该分数而非高一分

**评分规则**：
- 相对目标期刊标准评分（Nature 比区域期刊更难）
- 非所有 Agent 对所有章节评分（如 Agent C 可能跳过标题）
- 评分矩阵中未审阅的章节用 `null`
- 每条评语以 `[Score: X/10]` 结尾并附一句理由
